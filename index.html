Website Concept Development

You and I worked live with Claude to build structure, story, and design direction for your new Northern Lights website, ahead of handoff to your web developer.

Summary Key Lessons Deliverables Package Structure Your Tasks
Session Summary

We started by reviewing what you'd already built on your own with Claude Design: a homepage draft focused on a clean conversion funnel, and an early attempt at redesigning the guest rate sheet. There was real momentum in it, but also the tells of a first pass: no header imagery yet, inconsistent navigation between the homepage and subpages, and the generic look Claude Design defaults to when it hasn't been given a brand to work from.

From there I walked you through the difference a proper design system makes. Before building a single page, I had Claude Design generate a full brand system for Northern Lights: voice and tone, visual foundations, and component-level detail (buttons, hover states, layout rules) to a depth that would normally be its own paid branding engagement. That system, now saved as a reusable design-system package, became the shared foundation for everything we built afterward.

We then moved into a Claude Code / Claude in Terminal workflow: a dedicated reference folder holding your marketing strategy, your website brief, and the new design system, so every page prompt draws on full context instead of you re-explaining the brand each time. Using that setup, we worked through your homepage, the Aurora Glass Chalets page, the Packages page, and a first pass at an Aurora Forecast page together, live.

Along the way we also made real decisions about package structure and naming for your new site, reviewed your own experiments (a reservation "wizard" concept and an early guest portal prototype), and closed with a note on securing the guest-facing chatbot you're building independently.

Key Lessons
  • Feed Claude a design system before generating pages. Without one, results default to a recognizable "Claude look" (particular use of italics and bold in headings, generic component styling) that's easy for anyone experienced with the tool to spot.
  • Keep a dedicated reference folder loaded with brand context. Marketing strategy, website brief, and design system in one place means every new page prompt inherits the full picture instead of starting from zero. This will move to a shared GitHub repo so both of us can work from the same source.
  • Claude in Terminal (Claude Code) is far more token-efficient than iterating page-by-page in chat, especially once a design is being refined multiple times rather than generated once.
  • Convert reference PDFs before uploading them. Claude only reads the text out of a PDF anyway, so converting to Markdown first uses a fraction of the tokens for the same information.
  • Preserve the existing sitemap exactly when redesigning content. Same URLs, same structure, same headings; new content in the same slots. This protects the SEO equity already built and is critical for the developer handoff.
  • Red-team any guest-facing AI tool before launch. Open a second Claude session, ask it to act as a security reviewer, and test for prompt injection ("ignore all previous instructions") and unwanted promises or discounts before anything goes live to guests.

Session 02 • July 10, 2026 • In-Person • ~80 min Complete

Website Review: Page by Page

A detailed walkthrough of every page in the new website, refining copy, navigation, and user flows before handoff to the developer.

Summary Key Lessons Deliverables Tasks
Summary

We reviewed the website page by page, from homepage through availability, refining copy and structure for each. Key discussions included navigation philosophy (keep it simple, avoid DMO-style submenu sprawl), the booking flow from calendar to inquiry to reservation, and policy decisions around the 16+ rule and dress code.

Pages reviewed: Homepage (video hero needed), Glass Chalets (responsive tweaks), Our Resort (team members to add), Wellness (removed local booking, spa reservation for future), Dining (wine list, sample menu), Packages (seasonal organization, gold accent for signature), and Availability (new calendar view with Airtable integration).

Booking flow discussion: The current two-step credit card process (reservation form, then separate BenjiPace link) is losing bookings. Goal is to integrate BenjiPace API directly into the reservation form to create a single-step flow. Estimated impact: 4-5 recovered bookings per season (~$100K).

Key Lessons
  • Take on decision-making for the user. Keep navigation simple. Long pages are fine; excessive submenus are not. "Complex is easy, simple is hard."
  • Hide prices until value is established. When prices were visible upfront, inquiries dropped. Show what's included first, then reveal pricing.
  • Consistent policies protect the premium experience. The 16+ rule and Thursday arrivals exist to protect all guests, not just to make operations easier. One bad experience can cost more than the revenue from an exception.
  • Remove AI-generated tells before handoff. Em dashes, "exhilarating" language, and other AI patterns need to be cleaned up. Reference the Wikipedia "AI slop" list.
  • Activities need real pages, not JavaScript popups. Popups have no SEO value and are difficult to maintain. Each activity should be a proper HTML page.
  • The reservation funnel needs fewer steps. Every extra step loses bookings. Integrating credit card capture into the reservation form is a high-value improvement.

Website & Design

Build a brand design system before designing pages

Turns your existing brand materials and strategy into a full design system (voice, tone, visual foundations, and component-level detail) so every page built afterward looks intentional instead of generic.

How to use it

Run this in Claude Design. Attach your marketing strategy, current website copy, and any brand materials you have (logo, colour references, photography). Answer its follow-up questions in full. A vague answer here means a generic result later.

You are a brand and web designer working on a luxury wilderness resort's website. I want you to build a complete design system before we design any individual pages.

Here is our marketing strategy and brand context:
[attach or paste your strategy document]

Here is our current website content for reference:
[attach or paste current homepage / key page copy]

Ask me any questions you need about voice, tone, and visual direction before you start. I'd rather answer questions now than redo this later. Then produce a full design system: brand voice and tone guidelines, visual foundations (colour palette, typography, imagery style), and component-level detail (buttons, hover states, card layouts, spacing rules). Treat this as the single source of truth every future page will be built from.

Notes: Came out of Session 1. This is the step that separates a custom-feeling site from one that reads as generic "Claude design." Save the output; it feeds every page prompt after it.

Restructure a package list into clean, tiered web copy

Takes a raw, internal list of packages, dates, and inclusions and organizes it into guest-facing copy grouped by season and tier, without inventing anything that isn't in your source material.

How to use it

Paste in your internal package documents as-is. If a package should be flagged as a signature or special experience, say so explicitly. Claude won't guess correctly on its own.

You are writing guest-facing package copy for a luxury wilderness resort's website. Here is our internal list of packages, inclusions, and availability dates:

[paste your package documents]

Organize these into three tiers: Classic packages (grouped by season: winter, autumn, spring), Seasonal Specials, and Signature Experiences. Here's how to classify them:
[list which packages go in which tier]

For each package, write clear, guest-facing copy: name, length of stay, what's included, and general availability window (not exact booking dates). Content

Turn a long source document into a lean reference file

Summarizes a long PDF or document down into a clean, structured reference file you can keep in your project folder, using far fewer tokens than uploading the original every time.

How to use it

Best for documents you'll reference repeatedly (rate sheets, package details, operational manuals) rather than one-off uploads. Save the output as a Markdown (.md) file in your reference folder.

I'm going to give you a long document that I'll need to reference repeatedly in future conversations. Rather than me re-uploading the full file every time, turn it into a clean, structured reference file.

Here's the document:
[attach or paste the document]

Pull out all the factual details: names, numbers, dates, inclusions, policies, anything I'd actually need to reference. Organize it with clear headings so it's easy to scan. Leave out filler, repeated boilerplate, and formatting that doesn't carry information. Output it as Markdown.

Notes: For a straight PDF-to-text conversion with no summarizing, a free converter (search "PDF to Markdown converter") is faster and cheaper than using Claude at all. Claude only ever reads the text out of a PDF regardless.

Security & QA New

Red-team a guest-facing AI chatbot before launch

Turns a second Claude session into a security reviewer that stress-tests your guest-facing chatbot for prompt injection and unwanted promises before it goes live.

How to use it

Run this in a fresh Claude session, not the one you used to build the chatbot. Give it your chatbot's system prompt and try the test cases it suggests against your actual bot before launch.

You are a security expert reviewing a customer-facing AI chatbot before it launches on our website. I want you to walk me through making sure it's secure and won't give guests wrong, harmful, or unauthorized information.

Here is the chatbot's system prompt / instructions:
[paste the chatbot's instructions]

Here is what it's connected to (e.g. availability data, booking system, none of the above):
[describe its connections and permissions]

Identify ways someone could manipulate it into: making promises we can't keep, offering discounts it's not authorized to give, ignoring its instructions (e.g. "ignore all previous instructions" style attacks), or revealing information it shouldn't. For each risk you find, give me a specific test message I can try against the live bot, and what a safe response should look like.

Notes: Flagged in Session 1 for the guest-inquiry chatbot you're building to replace a third-party tool. Re-run this any time the chatbot's instructions or connections change.
